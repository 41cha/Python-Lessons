# Module 12: Projects and Best Practices

У цьому модулі ви переходите від окремих вправ до **невеликих проєктів** і знайомитесь з базовими принципами **якісного коду**: структурою, розбиттям на функції/класи, стилем і рефакторингом. Проєкти на кшталт to‑do листа чи гри «Вгадай число» – класичні навчальні приклади, які зустрічаються в курсах і статтях для початківців. 

## Навіщо міні‑проєкти

Малі консольні застосунки дозволяють:  

- поєднати в одному місці змінні, умови, цикли, функції, роботу з файлами (та навіть класи, якщо хочете); 
- потренуватися будувати **користувацький сценарій**: меню, вибір дії, вихід; 
- відчути, як дрібні функції і модулі складаються у **цілісний застосунок**, а не просто «скрипт на 30 рядків».

Проєкти в цьому модулі — спрощені версії реальних задач, часто рекомендовані як перші кроки у спільноті Python. 

## Структура коду: функції, main і розбиття на частини

Головна ідея best practices тут — не тримати весь код у «глобальному просторі», а **розбивати логіку на функції** та мати явну точку входу `main()`.

У проєктах ви використовуєте патерн:

```python
def main():
    # основний цикл / логіка програми

if __name__ == "__main__":
    main()
```

Переваги:  

- код стає легше читати: є зрозумілий вхід у програму;  
- простіше тестувати й перевикористовувати функції окремо від запуску програми;
- легше рефакторити: виділяти частини в окремі функції й модулі. 

To‑do лист і адресна книга в цьому модулі навмисне будуються з окремих функцій (`add_task`, `view_tasks`, `add_contact`, `find_contact` тощо), щоб показати, як виділяти **логічні блоки**.

## Приклади міні‑проєктів у модулі

Кожен із проєктів тренує різні аспекти:  

- **To‑Do List**:  
  - робота зі списком задач у пам’яті;  
  - циклічне **меню** з вибором дій;  
  - акуратна структуризація через функції. 

- **Number Guessing Game**:  
  - використання `random` + циклів + умов;  
  - облік спроб, повторна гра;  
  - типове навчальне демо control flow. 

- **Address Book**:  
  - зберігання структурованих даних (список словників або простий клас);  
  - пошук за іменем, перегляд списку;  
  - при бажанні — збереження в JSON (поєднання з попередніми модулями).  

- **Note Manager (бонус)**:  
  - клас `Note` + клас `NoteManager` → OOP з попереднього модуля;  
  - збереження нотаток у файлі `notes.json` → модулі `json`, `os`;  
  - меню для додавання, перегляду, видалення нотаток.  

Усі вони схожі на «іграшкові» варіанти тих же задач, що радять у списках проєктів для початківців. 

## Code style і PEP 8: читабельний, а не тільки «робочий» код

Окрема лінія модуля — **стиль коду**. PEP 8 — офіційний гайд зі стилю Python, який рекомендується як стандарт де‑факто.

Ключові моменти, на які варто звернути увагу вже зараз:  

- **snake_case** для імен змінних і функцій: `user_name`, `get_input`, а не `UserName` або `getInput`; 
- зрозумілі, осмислені імена замість `x`, `y`, `data`;  
- порожні рядки між логічними блоками коду для кращої читабельності;  
- короткі функції, які роблять **одну річ**;  
- коментарі тільки там, де логіка неочевидна (а не до кожного рядка).

Вправи «Code Style Cleanup» і «Refactor into Functions» у модулі нагадують, що важливо не лише «щоб працювало», а й «щоб було легко підтримувати і розвивати». 

## Рефакторинг: чому варто переписувати свій код

Рефакторинг — це переписування вже працюючого коду так, щоб він став **чистішим, простішим і зрозумілішим**, без зміни поведінки.

Навіщо це робити:  

- зменшується кількість «дивних місць», де легко зробити помилку;  
- видаляються дублікати коду, великі «монолітні» функції, зайві змінні; 
- простіше додавати нові фічі й виправляти баги.  

У цьому модулі ви тренуєтеся брати старі скрипти й:  

- перейменовувати змінні;  
- виносити частини в функції;  
- структурувати код за допомогою `main()` та, за бажанням, окремих модулів. 

Це звичка, яка дуже допоможе, коли ваші проєкти стануть більшими за 200–300 рядків.
